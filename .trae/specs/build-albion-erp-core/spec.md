# Albion Online ERP 核心后端 Spec

## Why
当前仓库仅有用户模块的示例性代码，尚未覆盖 PRD 中定义的登录注册、第三方绑定、游戏账号绑定审核、角色切换、工会注册与成员管理、战斗统计与补装统计等核心能力。需要基于现有 pnpm Monorepo、Cloudflare Workers、Hono、Drizzle ORM 架构，补齐第一阶段可落地的后端能力，为后续前端接入和后台运营提供稳定接口。

## What Changes
- 新增平台用户基础数据模型，覆盖邮箱登录注册、忘记密码、第三方账号绑定、当前角色切换所需字段与状态。
- 新增游戏账号绑定申请与人工审核流，支持每个用户最多绑定 10 个游戏账号，并记录绑定 token、服务器、审核状态。
- 新增工会注册、工会角色权限、成员管理的基础数据模型与 API。
- 新增战斗记录导入、补装 session、补装记录、审批流水、自动审批规则等业务基础能力。
- 新增共享 Zod Schemas 与 Hono 路由，并统一挂载到 API 入口导出 RPC 类型。
- 为第一阶段人工审核场景预留管理后台接口，不实现游戏内自动校验。

## Impact
- Affected specs: 用户认证、第三方绑定、游戏账号绑定、当前角色切换、工会注册、工会权限、工会成员、战斗统计、补装统计、人工审核后台
- Affected code: `packages/db/src/schema/*`、`packages/shared/src/schemas/*`、`packages/shared/src/index.ts`、`apps/api/src/routes/*`、`apps/api/src/index.ts`

## ADDED Requirements
### Requirement: 平台用户认证与恢复
系统 SHALL 提供邮箱注册、邮箱登录、忘记密码重置与第三方登录接入所需的后端数据结构和 API。

#### Scenario: 邮箱注册成功
- **WHEN** 游客提交合法邮箱、验证码与密码完成注册
- **THEN** 系统创建平台用户并标记邮箱已验证

#### Scenario: 忘记密码成功
- **WHEN** 已注册用户提交合法邮箱验证码与新密码
- **THEN** 系统更新密码凭据并允许其重新登录

### Requirement: 第三方账号绑定
系统 SHALL 支持 Kook、Discord 账号与平台用户的绑定、换绑、解绑，并保证每个用户在每个平台最多绑定一个账号。

#### Scenario: 绑定第三方账号成功
- **WHEN** 已登录用户绑定一个未被占用的 Discord 或 Kook 账号
- **THEN** 系统建立唯一绑定关系并返回最新绑定列表

#### Scenario: 换绑第三方账号成功
- **WHEN** 已登录用户在同一平台重新绑定另一个账号
- **THEN** 系统替换旧绑定并保留审计所需状态字段

### Requirement: 游戏账号绑定申请与人工审核
系统 SHALL 支持用户提交游戏账号绑定申请，记录服务器、游戏 ID、绑定 token、审核状态，并限制每个用户最多 10 个绑定账号。

#### Scenario: 创建绑定申请成功
- **WHEN** 已登录用户提交游戏 ID 与服务器，且当前未超过 10 个账号上限
- **THEN** 系统生成唯一绑定 token 并创建待审核申请

#### Scenario: 管理员审核通过
- **WHEN** 管理员人工确认用户已在游戏个性签名中填写绑定 token 并执行通过操作
- **THEN** 系统将申请状态改为已通过，并将该游戏账号标记为用户可用角色

### Requirement: 当前角色切换
系统 SHALL 允许用户在已绑定且已审核通过的游戏角色中切换当前角色，并在接口层返回当前角色上下文。

#### Scenario: 切换当前角色成功
- **WHEN** 用户选择一个属于自己的已审核通过角色
- **THEN** 系统更新当前角色指针，后续接口可读取当前角色上下文

### Requirement: 工会注册与人工审核
系统 SHALL 支持用户提交工会注册申请，记录工会全称、服务器、绑定 token、审核状态，并在审核通过后为提交人分配默认管理员角色。

#### Scenario: 提交工会注册成功
- **WHEN** 已登录用户提交工会全称与服务器
- **THEN** 系统生成工会绑定 token 并创建待审核工会

#### Scenario: 审核通过后初始化管理员
- **WHEN** 管理员审核通过工会注册
- **THEN** 系统创建默认不可删除的管理员角色，并将提交人加入工会管理员成员关系

### Requirement: 工会角色权限与成员管理
系统 SHALL 提供工会角色、权限、成员关系管理能力，支持基于游戏账号或平台账号的成员绑定方式。

#### Scenario: 基于游戏账号添加成员
- **WHEN** 工会管理员为某工会添加一个已绑定且服务器匹配的游戏账号
- **THEN** 系统创建成员关系并允许配置角色与权限

#### Scenario: 基于平台账号添加成员
- **WHEN** 工会管理员按平台账号添加成员
- **THEN** 系统允许跨服务器关联该平台账号，不受游戏账号服务器限制

### Requirement: 战斗记录导入与展示
系统 SHALL 支持工会导入外部战斗记录，并在战斗记录中展示与补装 session 的关联状态。

#### Scenario: 导入战斗记录成功
- **WHEN** 具备权限的成员导入外部战斗数据
- **THEN** 系统创建战斗记录，并允许后续关联到补装 session

### Requirement: 补装 Session 与补装审批
系统 SHALL 支持从战斗死亡记录生成补装 session、补装记录、审批状态流转与审批流水。

#### Scenario: 生成补装 session 成功
- **WHEN** 有权限成员勾选多条战斗记录创建补装 session
- **THEN** 系统创建 session 并为每条死亡记录生成一条待处理补装记录

#### Scenario: 手动审批补装记录
- **WHEN** 审批人更新补装记录状态并填写备注
- **THEN** 系统写入新的审批流水并返回最新状态

### Requirement: 自动审批规则
系统 SHALL 提供可配置、可热插拔的自动审批规则管线，用于在生成补装记录时执行自动拒绝或自动审核。

#### Scenario: 自动规则命中拒绝
- **WHEN** 新补装记录触发一条拒绝规则
- **THEN** 系统自动将该记录标记为已拒绝并记录规则执行结果

### Requirement: 补装装备汇总与箱子分配
系统 SHALL 支持按 session 和按成员聚合补装装备信息，并为工会成员维护箱子坐标分配。

#### Scenario: 查看装备汇总
- **WHEN** 有权限成员查看某个补装 session 的装备统计
- **THEN** 系统按 P 级聚合装备需求，并同时支持按成员分组返回

#### Scenario: 配置箱子坐标
- **WHEN** 工会管理员为成员设置箱子位置
- **THEN** 系统保存 x-y 坐标并可供补装模块查询

## MODIFIED Requirements
### Requirement: 用户模块示例接口
现有用户模块示例 SHALL 从演示性质的邮箱绑定与游戏角色绑定接口，扩展为符合 PRD 的正式用户认证、账号绑定、角色切换与后台审核能力。

## REMOVED Requirements
### Requirement: 游戏内自动校验
**Reason**: PRD 明确第一阶段先采用人工审核，不实现游戏内自动化绑定验证。
**Migration**: 通过管理后台审核接口和绑定 token 字段承接验证流程，后续若接入自动校验，再在现有申请状态流基础上扩展。
